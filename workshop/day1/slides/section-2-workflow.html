Slide 2.6

Turning Ideas into Specs

create-prd.mdc

Vague idea → AI asks 3–5 questions → You answer "1A, 2C, 3B" → Structured PRD
⚙ What it does
  • Takes a vague feature idea as input
  • AI asks 3–5 clarifying questions with lettered options
  • Produces structured PRD: overview, goals, user stories, requirements
  • Saves to tasks/prd-[name].md
✶ Why it exists
  • Forces you to think through the feature before any code
  • Surfaces edge cases you'd discover mid-implementation
  • Gives AI enough context to produce good code later
⚠ Without it
  • AI builds based on assumptions
  • Missing requirements discovered 3 tasks deep
  • Rework, frustration, "AI sucks" feelings

Slide 2.10

How Things Go Wrong

And how the system prevents it

1 No PRD
AI builds the wrong thing. Assumptions everywhere. Discover misalignment after hours of work.
create-prd forces spec-first thinking
2 Tasks Too Big
AI loses context mid-task. Output quality degrades. Half-finished work that's hard to salvage.
generate-tasks enforces small, focused chunks
3 No Checkpoint
One bad task cascades. No clear rollback point. Everything is tangled together.
Checkbox tracking + atomic commit per task
4 No Tests
Can't verify if anything actually works. Bugs hide until production. No confidence to ship.
Test-first culture baked into task execution
5 Wrong Direction
Wasted effort building the wrong architecture. Days lost before anyone notices.
Two-phase "Go" gate catches it early
